# Type Rogue 生き物スプライト

このリポジトリへ生き物を安全に追加する。候補を決める前に生成や実装を始めない。

## 1. 現状を調査する

リポジトリルートで次を実行する。

```bash
node .claude/skills/type-rogue-creature-sprites/scripts/inventory-creatures.mjs
```

次も読む。

- `src/domain/fish.js`
- `concept_art/sprites/README.md`
- `concept_art/sprites/` と `public/sprites/`
- 配置候補のステージ定義と海域名
- [references/creature-roster.md](references/creature-roster.md)

種ID・表示名・実在分類・モチーフ・輪郭・配色・海域を比較する。完全一致だけでなく、同じ分類群や同じ魚影の意味的重複も調べる。

## 2. 最初に候補を提案して承認を待つ

ユーザーが実装を明示していても、対象が未確定なら最初に2〜4候補を提案する。通常は3候補を出し、1候補を推奨する。

各候補について次を示す。

- 実在名または創作名
- 実在・実在モチーフ・創作の分類
- 海域と配置理由
- 既存生き物との重複監査結果
- 輪郭、主要色、4コマの動き
- 水槽での泳ぎ（泳ぐ層 上/中/底、群れるか、大きさの強弱）。生態に沿って決める。詳細は手順7を参照
- 現在と追加後の実在生物比率
- 新規種、既存種の見た目差し替え、色柄違いのどれか

重大な重複があれば候補から外す。意図的な近縁種や色柄違いなら、既存種と見分ける方法を明記する。

候補表のあとに推奨理由を短く述べ、ユーザーの承認を待つ。承認前に画像生成、アプリデータ変更、公開素材の上書きを行わない。ユーザーが同じ依頼内で特定種を明示し、そのまま実装するよう明確に承認している場合だけ、この停止を省略できる。

## 3. 実在生物比率を守る

[references/creature-roster.md](references/creature-roster.md) の分類規則を使う。

- `FISH_SPECIES` に登録された収集対象全体の70〜80%を実在生物にする方向で提案する。スプライト未設定種も分母へ含める。
- 実在種または実在する分類群を実名で表したものだけを「実在」に数える。
- 創作名、実在モチーフ、判定不能は実在比率の分子へ入れない。
- 現在比率が70%未満なら、回復するまで実在生物を優先する。
- 小さな追加単位で直ちに70%へ到達できない場合は、今回と次の追加後の見込みを併記する。
- 80%を超えた場合は創作種を候補へ戻せるが、海域学習や収集体験に理由があるものだけにする。
- 必要ならスプライト対応種だけの比率も補助値として示すが、全体比率の代わりにしない。

## 4. 承認後にアート仕様を確定する

実在生物では、体形、模様、ひれ、目、主な生息域を信頼できる水族館・博物館・研究機関などで確認する。創作要素と実在の特徴を混同しない。

[references/sprite-spec.md](references/sprite-spec.md) を読み、次を確定する。

- 右向きの読みやすい輪郭
- 既存種と区別できる色と柄
- 4フレームの変化
- 生き物に含まれないクロマキー色
- 対応する種ID、海域、ステージ、レア度

色違い・柄違いは同じ魚影を再利用してよい。ただし、名前、配色、柄のうち最低2つで既存個体と区別する。

## 5. 画像を生成する

利用可能な画像生成スキルまたは画像生成ツールの指示を先に読む。既存の完成スプライト2〜3点を画風と配置の参照に使い、1素材ずつ生成する。

画像生成機能が利用できない場合は、承認済みのアート指示書まで作成して停止し、ユーザーへ原画の提供を依頼する。別の手段で画像を捏造しない。

プロンプトへ必ず含める。

- 同一個体を横一列4セル
- 全フレーム右向き、同じ大きさ、同じ基準線
- 休止、ひれまたは尾の変化、反対側の変化、泡または小さな特徴
- 既存素材と同じドット密度、輪郭、限定色
- 単色クロマキー背景
- 文字、UI、区切り線、景色、余分な生き物を入れない

## 6. 透過・整形・目視確認を行う

原画を直接縮小し、拡大済みの小画像を素材にしない。

```bash
python3 .claude/skills/type-rogue-creature-sprites/scripts/prepare-sprite.py \
  INPUT.png OUTPUT.png --key-color auto

python3 .claude/skills/type-rogue-creature-sprites/scripts/validate-sprite.py \
  OUTPUT.png
```

透過結果を原寸で開き、次を目視する。

- 背景色の縁が残っていない
- 体色や細いひれが欠けていない
- 各セルに1匹だけいる
- 4フレームで同じ個体に見える
- 64 CSS px前後でも輪郭が読める
- 泡コマで本体が大きく横移動しない

問題があればクロマキー色または閾値を変えて再生成・再処理する。破損した輪郭をそのまま採用しない。

## 7. アプリへ接続する

承認された最終素材を次の両方へ同名で置く。

- `concept_art/sprites/<name>-strip.png`
- `public/sprites/<name>-strip.png`

`src/domain/fish.js` の種データへ `rarity` と `sprite` を設定する。スプライトURL、向き、現在フレームを捕獲個体やセーブへ保存しない。新しい実在種を追加するときは安定した種IDを作り、既存IDを別種へ流用しない。

### 水槽での泳ぎ・見え方の属性

水槽内の遊泳挙動は種データの次の属性で決まる（定義とデフォルトは `src/domain/fish.js` 冒頭コメントが正）。生き物の生態に合わせて追加時に設定する。省略時は「群れない・中層・等倍・巡航」になる。

- `school: true` … 同種で群れる。小さくおとなしい魚（スズメダイ、テンジクダイ等）に付ける。大型や単独性（アンコウ、エイ、タコ等）には付けない。群れる魚は共通の中心へ集まって泳ぐ。
- `depth: "top" | "mid" | "bottom"` … 泳ぐ層。既定は `"mid"`。底生（ハゼ、エイ、ウツボ、チンアナゴ等）は `"bottom"`、浮遊する種（クラゲ、イカ等）は `"top"`。
- `scale: 数値` … 水槽での大きさ倍率。既定 `1`（＝最小）。大型魚だけ大きくして強弱を出す（例: エイ・ウミガメ `1.5`、アンコウ `2`）。当たり判定にも反映されるので枠外へは出ない。
- `movement: "drift"` … ふわふわ漂う動き。クラゲ、タツノオトシゴなど遊泳力の弱い種に付ける（既定は巡航）。
- 見た目の基礎: `shape`（`round`/`long`/`tall`）、`color`、`accent` はスプライト未設定種のCSS描画にも使うので必ず設定する。

未設定種のCSS魚フォールバック、アプリの動き軽減、`prefers-reduced-motion` を維持する。`concept_art/sprites/README.md` の割り当て、仕様、生成概要も更新する。

## 8. 検証とQAを行う

```bash
node .claude/skills/type-rogue-creature-sprites/scripts/inventory-creatures.mjs
python3 .claude/skills/type-rogue-creature-sprites/scripts/validate-sprite.py \
  concept_art/sprites/<name>-strip.png \
  --mirror-dir public/sprites
npm test
npm run build
git diff --check
```

QA担当を使える場合は、実装担当とは別の担当へ読み取り専用監査を依頼する。少なくとも画像寸法・透過・4セル、重複、海域、捕獲可能性、セーブ非混入、フォールバック、動き軽減を確認する。QA担当がいない場合は同じ項目を別の確認パスとして実行する。

## 担当の境界

- 主担当: 候補提案、実在比率、承認管理、統合
- アート担当: 承認済み仕様から原画と差分案を作る
- 実装担当: 整形、魚種設定、表示接続、ドキュメント
- QA担当: 生成・実装担当から独立して監査する
- 問題担当: 生き物名を教材へ追加する依頼がある場合だけ語彙と例文を扱う

並行作業では担当範囲を分離する。問題データを生き物追加のついでに変更しない。
