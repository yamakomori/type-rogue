# UI/UX 設計

## 基本方針

子どもに「何をすればよいか」を読ませない。画面ごとに主な行動を一つに絞り、最初の問題までを最短にする。初めての子には説明より実演を使う。

## 画面構成

| 画面 | 主役 | 主な操作 | 常に見せないもの |
| --- | --- | --- | --- |
| はじめる | 相棒と開始 | 「はじめる」 | 設定、通貨、長い説明 |
| 水槽 | 育ったコレクション | 「海へ出かける」 | 達成率、複雑な世話 |
| 海図 | 次の海 | 海を選ぶ | 全海域の細部、成績表 |
| タイピング | 問題と次のキー | キーボード入力 | 店、通知、残高 |
| 結果 | つれた魚 | 「海図へ」「もう1回」 | 詳細統計、広告的誘導 |
| お店・着せ替え | 相棒とアイテム | 試着、交換、装備 | 複数通貨、期限、確率 |
| 設定 | 音・動き・配列 | 切替 | 学習進行をリセットする操作 |

## 初回導線

```text
開始
  → 「F と J のぽっちをさわってみよう」
  → 1キー問題
  → 2キー問題
  → 最初の魚をつる
  → 水槽をひと目見る
  → 最初の海へ
```

オンボーディングはいつでも終了でき、次回の開始時に再表示しない。キーボードが接続されていない場合は開始前にわかりやすく案内し、操作不能な画面を見せない。

現行MVPでは、新しい冒険だけに「F と J のぽっちをさわってみよう」を表示する。3問を終えて最初の魚をつった後は、水槽を見るボタンを一度だけ主ボタンとして出す。途中で「海図をみる」を選んでも、次回はこの案内を繰り返さない。

最初の釣果ダイアログは、魚名・短い一言・「水槽をみる」だけに限定する。メダル、新しい海、再プレイなどの情報は記録だけ行い、この画面には出さない。

## タイピング画面

優先順位は次の通り。

1. 表示文（読みを含む）
2. 現在位置と次のローマ字
3. 次のキーと指ガイド
4. 小さな相棒・近づく魚影
5. 一時停止・終了

- 問題は画面中央付近に固定し、レイアウトを打鍵ごとに動かさない。
- ローマ字ヒントは、入力中の別綴りに合わせて更新する。
- 画面キーボードは初期表示し、設定で小さくする・非表示にするを選べる。
- ミス時も問題文の色を大きく変えず、次キーの案内を保つ。
- 途中終了は常に可能とし、確認は「続ける」「地図へ」の2択だけにする。

## 結果画面

1回のプレイ結果は、2秒以内に理解できる情報量にする。

- 最初に「つれた：ぽっちハゼ」のような獲得物を見せる。
- 次に一言だけ成長を伝える。
- その下に、同じ大きさの「海図へ」と「もう1回」を置く。
- 正答率、WPM、順位、連続日数は初期画面には置かない。

## アクセシビリティ

- キーボードだけで全操作でき、フォーカス位置を明確にする。
- 読み上げ用ラベルで問題進捗・次キー・結果を伝える。打鍵ごとの過剰な読み上げは避ける。
- 日本語/US配列の選択と確認画面を用意する。記号問題は配列確定後に出す。
- 文字サイズを拡大しても問題文、ヒント、終了ボタンが隠れない。
- 高コントラスト・動き軽減・音オフを設定で提供する。

## 子どもレビューの観点

8〜10歳のテストでは、説明後ではなく初見で以下を観察する。

- 次に何をすればよいか、10秒以内にわかるか。
- ミス後に自分で入力を再開できるか。
- 1プレイ後に「もう1回」と思う理由が言えるか。
- 使った言葉が読めるか、怖い・恥ずかしい・急かされると感じないか。

## 評価

| 観点 | 設計上の扱い |
| --- | --- |
| 学習効果 | 次の一打に集中できる視線設計と、邪魔しないヒント |
| 継続率 | 開始から報酬まで短く、終了も気持ちよくできる |
| UX | 画面ごとに主行動を一つにし、初見操作を検証する |
| 実装コスト | まず6画面と共通のタイピング画面に限定する |
