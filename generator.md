# 問題生成エンジン設計

## 目的

カリキュラムと個人の苦手をもとに、短く・重複感が少なく・入力可能な問題を生成する。無限生成を目的にランダム文字列を作らず、意味のある語彙データから安全に組み立てる。

## 入出力

```ts
type ProblemRequest = {
  stageId: string;
  profile: SkillProfile;
  mode: "lesson" | "review" | "treasure";
  maxSeconds: 30 | 45 | 60;
  seed?: string;
};

type Problem = {
  id: string;
  stageId: string;
  text: string;
  kana: string;
  acceptedInput: "ja-romaji" | "direct";
  targetKeys: string[];
  introducedKeys: string[];
  tags: string[];
  estimatedKeystrokes: number;
  difficulty: 1 | 2 | 3 | 4 | 5;
};
```

`kana` は入力判定に渡す読みであり、表示文と分離する。漢字の表示を使う場合も、読みはコンテンツデータに明示して自動変換へ依存しない。

## コンテンツデータ

問題の素材は手書きの語彙パックとして管理する。

```json
{
  "id": "animal-cat-001",
  "text": "ねこ",
  "kana": "ねこ",
  "tags": ["animal", "noun", "calm"],
  "minStage": "S09",
  "targetKeys": ["n", "e", "k", "o"],
  "features": ["seion"],
  "difficulty": 1
}
```

素材には以下を必須とする。

- 表示文と読み、使用キー、拍数、ローマ字特性（`sokuon`、`hatsuon`、`youon` など）
- 対象年齢に適したタグと、感情の強さを示す安全タグ
- `minStage` と、出してはいけないキー・記号
- 人名、住所、学校、身体的特徴、恐怖・暴力・課金を連想させる内容を含まないこと

## 生成手順

1. ステージから使えるキー・特性・問題長を決める。
2. プロフィールから苦手キー、新規キー、直近出題を取得する。
3. 素材を必須条件で絞る。
4. `新規 15% / 苦手 35% / 既習 50%` を基準に重み付き抽選する。
5. 直近10問と同じ語、同じ先頭2文字、同じタグが3回続く候補を外す。
6. ローマ字判定器で `kana` を検証し、推定打鍵数と時間が上限内か確認する。
7. 不適合なら最大20回再抽選し、失敗時は静的な安全問題にフォールバックする。

現在のMVPでは、候補から通常どおり重み付きで3問を選びつつ、復習対象キーがある場合は最初の1問が必ずそのキーを含むようにする。抽選関数は注入可能にし、選択規則をテスト可能に保つ。

## 問題の型

| 型 | 利用ステージ | 例 | ねらい |
| --- | --- | --- | --- |
| key | S00〜S02 | `f j f j` | 指とホーム位置 |
| pattern | S01〜S08 | `asdf` 相当の短い並び | 1指ずつの移動 |
| kana | S09〜S11 | `ねこ`、`きって` | ローマ字規則 |
| word | S12〜S13 | `あさのそら` | 意味のある反復 |
| sentence | S13〜S14 | 短く明るい一文 | 読む・打つ統合 |
| free | S15 | お題「きょう見つけたもの」 | 自発性 |

`key` と `pattern` は学習初期だけに使う。ランダムな英字列を長く出題しない。

## 難易度の制御

難易度は速度でなく、認知負荷を表す。

| 要素 | 難しくする条件 | 制限 |
| --- | --- | --- |
| 長さ | 推定打鍵数が増える | 初級は12打鍵以内 |
| キー移動 | 指・手の移動が増える | 新規キーは1問2種まで |
| ローマ字特性 | っ・ん・拗音・濁音を含む | 特性は初登場時1種のみ |
| 語彙 | 長い語・漢字・文脈依存 | 読めない語を避ける |
| 記号 | 句読点・長音・空白 | S14 まで必須にしない |

## 重み付け

各候補の重みは、以下の積で決める。値の絶対値より比率を調整可能にする。

`stageFit × weaknessFit × spacingFit × noveltyFit × varietyFit`

- `stageFit`: 現在の使用可能キーだけで打てるか。
- `weaknessFit`: 苦手キーを1〜2個含むと上げる。3個以上は下げる。
- `spacingFit`: 前回出題から2〜5問空いた復習を上げる。
- `noveltyFit`: 新規キー・特性を含む初回を上げる。
- `varietyFit`: 直近タグ・語頭・長さが重なると下げる。

## 検証とフォールバック

出題データの CI 検証では次を失敗扱いにする。

- 入力判定器が `kana` を処理できない。
- ステージで未解禁のキーを含む。
- 素材ID、表示文、読みが重複する。
- 必須メタデータ、安全タグ、読みがない。

実行時に候補が不足した場合、乱数で穴埋めせず、そのステージの検証済みフォールバック問題を出す。問題切れを子どもに見せない。

## 評価

| 観点 | 設計上の扱い |
| --- | --- |
| 学習効果 | 苦手キーを間隔反復し、一問に詰め込みすぎない |
| 継続率 | 意味のある短い問題と多様な題材で反復感を抑える |
| UX | 不正な問題・入力不能な問題を事前検証で排除する |
| 実装コスト | 語彙パック、重み付け、検証器を独立した純粋関数にする |
